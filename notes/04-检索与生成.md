# 04 · 检索与生成(M2)

M1 把文档灌进了向量库,M2 做的是**在线问答**那一半:用户问一个问题,系统检索相关资料、让 LLM 据此生成带出处的答案。

## 两条链路

| 链路 | 文件 | 做什么 |
|---|---|---|
| **Retrieve** | `src/rag/retrieve.py` | 问题 → 向量化 → 向量库检索 Top-K |
| **Generate** | `src/rag/generate.py` | Top-K 资料 + 问题 → 拼 Prompt → LLM → 答案 + 出处 |

两条链路由 `src/rag/pipeline.py` 的 `RagPipeline.ask()` 串起来,命令行入口是 `scripts/ask.py`。

## Retrieve:一个关键前提

```
retrieve(question, embedder, store, top_k, library=None)
```

**查询和入库必须用同一个 embedding 模型。** 否则问题向量和文档向量不在同一个语义空间,算相似度毫无意义。我们靠 `config.py` 里集中的 `embedding_model` 保证两边一致。

设计上**不设相似度阈值**:总是返回 Top-K,把"无关问题该不该答"交给 generate 的 prompt 兜底。阈值该设多少是个要到 M4 评估才能调的坑,现在拍脑袋定不准。

## Generate:防幻觉的三道闸

`build_prompt` 拼出的 prompt 里有三层约束(见 `generate.py` 的 `_SYSTEM`):

1. **只能依据【资料】回答** —— 限定信息来源。
2. **没有就说"根据现有资料无法回答"** —— 给模型一条合法退路,它就不必硬编。
3. **结构化分区**(资料/问题/回答)—— 让模型分清"给定事实"和"待答问题"。

### 出处用代码统计,不靠 LLM 自报

`answer()` 里的 `sources` 是**用 Python 遍历实际喂进去的 chunks 去重**得到的,不是让 LLM 在答案里自己写来源。因为 LLM 报来源本身也会幻觉(编个不存在的文件名)。用代码统计 → 出处 100% 真实。去重保持首次出现顺序。

## 架构:为什么这么分

- retrieve / generate / pipeline **全程只依赖接口**(Embedder / VectorStore / LLM),不碰 Ollama/Qdrant 具体类。
- "选哪个实现"的决定只发生在 `scripts/ask.py`(装配层)。
- 好处:测试能塞全 fake provider,0.5 秒跑完 27 个测试、不依赖 Docker/Ollama。

## M3 预告

`RagPipeline` 已经是三条链路的唯一汇合点,M3 的 FastAPI 直接复用它包成 HTTP 服务即可,不必重写装配逻辑。
