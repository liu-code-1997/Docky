# 07 · 检索优化(M5)

M4 诊断出瓶颈在检索层。M5 用消融实验(ablation)逐个尝试改进,每个都做成独立开关,
逐个评估看孤立贡献。三个方向:①markdown 切分 ②查询改写 ③重排。

本篇也是全项目**方法论教训最密集**的一篇——多个"想当然"被数字推翻。

## ① markdown 切分 + 噪声过滤

### 动机
baseline 用字符硬切(chunk_text),M4 验证过:问"路径参数",召回 Top-4 全是
"FastAPI Cloud 赞助""Conf 会议"等营销噪声块,正确文档进不来。假设:按 markdown
标题结构切 + 过滤噪声,能让正确正文浮上来。

### 实现(chunk_markdown)
- 按 `#`~`######` 标题切段,每段是语义完整的正文。
- 过滤两类垃圾:①噪声段(标题/正文含 sponsor/FastAPI Cloud/Conf/Deploying 等);
  ②HTML 样板段(按"标签密度"判定:标签+图片数相对实义字符过多即丢弃)。
- 超长段退回字符窗口切。

### 三个被数字推翻的"想当然"

**坑A:给每块加标题前缀,以为能补上下文,实测反而拉低命中。**
最初每块前缀标题链(如 `FastAPI > Path Parameters`)。结果 hit@4 从 baseline 的
50% 掉到 25%。cosine 探针发现:文档标题含高频词 "FastAPI",导致 index.md 的碎块
对任何含 FastAPI 的查询都虚高相似度(0.72+),霸占 Top-4。**去掉前缀后回升到 38%。**
教训:标题前缀是双刃剑,当标题含高频词、且查询跨语言(中文问/英文档)时弊大于利。

**坑B:设计文档被灌进知识库。**
spec 写在 docs/superpowers/specs/,而 ingest 灌整个 docs/,把项目自身文档也向量化了。
解法:把 specs 移出 docs/(docs/ 回归"纯知识库"语义)。教训:知识库目录不能混入项目文档。

**坑C(最重要):样本太少,分数差异是噪声,差点据此下错结论。**
在 8 条样本上 char=50% vs markdown=38%,险些判定"markdown 更差、放弃"。
把评估集扩到 18 条可答样本后,结论**反转**:markdown=44% > char=39%。
8 条时每条命中值 12.5%,一两条抖动就淹没真实差异。
教训:**样本少时别信分数;下结论前先把评估集做大。** 这正是评估驱动开发的核心纪律。

### ① 结论
markdown 切分(去前缀 + 噪声/HTML 过滤)在 18 条评估集上 hit@4 略优于 baseline
(44% vs 39%)。作为可选策略保留,`CHUNK_STRATEGY=markdown`(改后需重新 ingest)。
提升幅度不大,说明"切分"不是唯一瓶颈,②③仍有空间。
