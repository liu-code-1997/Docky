# 05 · 服务化(M3)

M2 把问答链路做成了能在命令行跑的 `RagPipeline`。M3 只做一件事:**给它套一层 HTTP 外壳**,让别的程序(前端、脚本、其它服务)能通过网络调用。

## 三个端点

| 方法 | 路径 | 作用 |
|---|---|---|
| POST | `/ask` | 提问,返回 `{text, sources}`(即 M2 的 Answer) |
| GET | `/libraries` | 列出库里有哪些 library 可供过滤 |
| GET | `/health` | 探活:能连上 Qdrant 就 200,否则 503 |

启动:`python scripts/serve.py`,然后访问 `http://localhost:8000/docs` 看交互式文档。

## 关键设计:工厂函数 + 依赖注入

`api.py` 不在模块顶层建一个全局 app,而是用 `create_app(pipeline, store)` 工厂:

```python
def create_app(pipeline, store) -> FastAPI: ...
```

- **测试**塞 FakePipeline / FakeStore,用 `TestClient` 秒跑,完全不碰 Ollama/Qdrant/网络。
- **生产**在 `scripts/serve.py` 里塞真实 provider。
- HTTP 层只做"翻译"——把请求转成 `pipeline.ask()` 调用,把结果转成 JSON。检索+生成逻辑一行没重写,全在 M2 的 pipeline 里。

这是 M1 就定下的依赖倒置思路一路贯穿到 M3 的结果:每一层只依赖接口,"选哪个实现"的决定始终推到最外层的 script。

## 两个 Web 层该有的边界

1. **输入校验**:`AskRequest` 用 `field_validator` 挡空白/缺失的 question,FastAPI 自动返回 422,轮不到 pipeline。脏输入挡在门口。
2. **探活语义**:`/health` 连不上 Qdrant 返回 503(Service Unavailable)——这是 k8s / 负载均衡器判断实例死活的标准状态码。

## `/libraries` 为什么值得单独做

它让 API **可自解释**:调用方不用翻服务器的目录结构,问一下就知道能按哪些 library 过滤。典型用途是前端做"选择文档库"下拉框,选项动态拉取而非写死——以后灌了新库,前端不用改代码。

代价是给 `VectorStore` 接口加了 `list_libraries()`(用 Qdrant 的 `scroll` 分页枚举 payload 里的 library,去重排序)。

## M4 预告

服务能跑了,但 M2 已经暴露:检索经常召回噪音块、答不出已有内容(notes/08 坑9)。M4 要做**评估**:建一个问题-答案集,量化"检索命中率""答对率",再针对性优化切分/检索。
