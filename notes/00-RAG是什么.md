# 00 · RAG 是什么

## 一句话

RAG = **R**etrieval-**A**ugmented **G**eneration,检索增强生成。
意思是:**先从你自己的资料里"检索"出相关内容,再把这些内容交给大语言模型(LLM)去"生成"答案。**

## 为什么需要 RAG

大语言模型(如 qwen、GPT)很强,但有两个硬伤:

1. **不知道你的私有/最新资料**
   模型只学过训练时见过的公开数据。你公司的内部文档、你正在用的某个库的最新文档,它没见过,自然答不准。

2. **会"幻觉"(hallucination)**
   当模型不确定时,它不会说"我不知道",而是一本正经地编一个看起来很像样、实则错误的答案。这在技术问答里非常危险。

RAG 的思路很朴素:**与其指望模型"记住"所有知识,不如在它回答前,先把相关资料"喂"给它。** 这样:
- 它能回答私有/最新内容(因为资料是你现给的)。
- 它更不容易幻觉(因为我们要求它"只根据给的资料回答,不知道就说不知道")。
- 答案可以**带出处**(我们知道资料来自哪个文件)。

打个比方:闭卷考试靠死记硬背(纯 LLM,容易记错),RAG 则是**开卷考试**——先翻到相关那几页(检索),再据此作答(生成)。

## 整体流程

RAG 分两个阶段。

### 阶段一:Ingest(灌数据,离线做一次)

```
你的文档(.md)
   │
   ├─ 加载:读出文本
   ├─ 切分(chunking):把长文档切成一小块一小块
   ├─ 向量化(embedding):每一块文本 → 一串数字(向量)
   └─ 存储:把"向量 + 原文 + 来源信息"存进向量数据库(Qdrant)
```

> 为什么要切分和向量化?因为"找相关内容"靠的是**向量相似度**——语义相近的文本,它们的向量在数学上也"靠得近"。详见 `02`、`03`。

### 阶段二:问答(在线,每次提问时做)

```
用户问题
   │
   ├─ 向量化:问题 → 向量
   ├─ 检索(retrieve):拿问题向量去向量库里找"最相近的 K 块"
   ├─ 拼 Prompt:把这 K 块资料 + 问题,组装成一段给 LLM 的话
   └─ 生成(generate):LLM 读完资料,给出带出处的答案
```

## 本项目的三条链路

我们把上面的流程拆成三个清晰、可单独测试的模块:

| 链路 | 做什么 | 对应代码(后续里程碑) |
|---|---|---|
| **Ingest** | 加载 → 切分 → 向量化 → 入库 | `ingest.py`(M1) |
| **Retrieve** | 问题 → 向量 → 检索 Top-K | `retrieve.py`(M2) |
| **Generate** | 资料 + 问题 → Prompt → LLM → 答案 | `generate.py`(M2) |

三条链路最后由 `pipeline.py` 串起来,再由 FastAPI 包成 HTTP 服务(M3)。

## 我们会亲手碰到的"坑"(这是学习重点)

第一版故意用最朴素的做法,好让这些真实问题暴露出来,再用评估(M4)看到它们的影响:

- 切分切多大?代码块被切断怎么办?(`03`)
- 检索不准:问"依赖注入"但文档写"Depends",找不到怎么办?(`04`)
- 幻觉:怎么让模型"不知道就说不知道"?(`05`)
- 我怎么知道我的 RAG 到底好不好?(`06`)
