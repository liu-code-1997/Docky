记录开发过程中实际遇到的报错和解法。每条:**现象 → 原因 → 解法**。

## M0 环境搭建阶段

### 坑 1:macOS 自带 Python 太旧,依赖装不上

- **现象**:`pip install -e ".[dev]"` 报错
  `AttributeError: module 'hatchling.build' has no attribute 'prepare_metadata_for_build_editable'`。
- **原因**:系统默认 `python3` 是 3.9.6(macOS 自带),版本过旧。项目要求 3.11+。
- **解法**:用 Homebrew 装的新版本。
  - 查找:`ls /opt/homebrew/bin/python3.*`(本机有 3.10/3.12/3.14)。
  - 用指定版本建 venv:`python3.12 -m venv .venv`。
- **教训**:Mac 上永远别用系统自带 python 做项目;认准 Homebrew 的版本。

### 坑 2:hatchling 找不到要打包的目录

- **现象**:`ValueError: Unable to determine which files to ship inside the wheel ...`
  `there is no directory that matches the name of your project (rag_docs)`。
- **原因**:项目名是 `rag-docs`,hatchling 默认去找名为 `rag_docs` 的包目录,但我们的包在 `src/rag`,名字对不上。
- **解法**:在 `pyproject.toml` 显式声明包路径:
  ```toml
  [tool.hatch.build.targets.wheel]
  packages = ["src/rag"]
  ```

### 坑 3:Docker Hub 连不上(国内网络)

- **现象**:`podman pull qdrant/qdrant` 报
  `pinging container registry registry-1.docker.io: Get "https://registry-1.docker.io/v2/": EOF`。重试稳定复现。
- **原因**:国内直连 Docker Hub(`registry-1.docker.io`)经常被阻断/超时。
- **解法**:给 Podman 配国内镜像加速器。在 machine 内写
  `/etc/containers/registries.conf.d/001-mirror.conf`,为 `docker.io` 加 mirror。
  ```bash
  podman machine ssh 'sudo tee /etc/containers/registries.conf.d/001-mirror.conf ...'
  ```

### 坑 4:镜像加速器域名已失效

- **现象**:配了中科大镜像 `docker.mirrors.ustc.edu.cn` 后报
  `dial tcp: lookup docker.mirrors.ustc.edu.cn: no such host`(DNS 解析不了)。
- **原因**:不少公共镜像站近年已关停或限制,域名不再解析。
- **解法**:先**测连通性再配**。在 machine 内逐个 curl 候选源的 `/v2/`:
  ```bash
  podman machine ssh 'for h in docker.m.daocloud.io docker.1panel.live docker.nju.edu.cn; do
    curl -s -m 8 -o /dev/null -w "%{http_code}" "https://$h/v2/"; done'
  ```
  本次可达的有:`docker.m.daocloud.io`、`docker.1panel.live`、`docker.nju.edu.cn`。
  把多个可达源都写进 mirror,Podman 会自动 fallback。
- **教训**:镜像源会过期,别照抄网上老教程的域名,先测再用。

### 坑 5:macOS 没有 `timeout` 命令

- **现象**:`timeout 60 podman pull ...` 报 `command not found: timeout`。
- **原因**:`timeout` 是 GNU coreutils 的命令,macOS 默认不带。
- **解法**:要么直接不加超时;要么 `brew install coreutils` 后用 `gtimeout`。

### 坑 6:Podman 没有 compose 提供者

- **现象**:`podman compose up -d` 报 `looking up compose provider failed`,
  找不到 `docker-compose` 或 `podman-compose` 可执行文件。
- **原因**:Podman 的 `compose` 子命令只是个转发器,需要本机另装 `podman-compose` 或 `docker-compose`。
- **解法**:学习项目里直接用 `podman run` 一条命令起容器,不依赖 compose:
  ```bash
  podman run -d --name rag-qdrant -p 6333:6333 -p 6334:6334 \
    -v "$(pwd)/qdrant_storage:/qdrant/storage:Z" qdrant/qdrant
  ```
  (`:Z` 是 SELinux 卷标,Podman 下挂载本地目录常用;Docker Desktop 下可省略。)

## M1 Ingest 阶段

### 坑 7:新版 httpx 的 mock Response 调 `raise_for_status()` 报错

- **现象**:测试里用 `httpx.Response(200, json={...})` 造假响应,被测代码调
  `resp.raise_for_status()` 时报
  `RuntimeError: Cannot call raise_for_status as the request instance has not been set`。
- **原因**:新版 httpx 的 `Response` 若没关联 `request` 对象,`raise_for_status()` 拿不到 request 就报错。
- **解法**:造假响应时补上 request:
  ```python
  httpx.Response(200, json={...}, request=httpx.Request("POST", url))
  ```
- **教训**:`raise_for_status()` 是对的(生产代码该保留),要改的是测试的 mock。

### 坑 8:qdrant-client 新版移除了 `.search()`

- **现象**:`self.client.search(...)` 报 `AttributeError: 'QdrantClient' object has no attribute 'search'`。
- **原因**:qdrant-client 1.10+ 用 `query_points()` 取代了已废弃的 `search()`。本机装的是 1.18。
- **解法**:改用新 API,注意返回值结构也变了(用 `response.points`):
  ```python
  response = self.client.query_points(
      collection_name=..., query=query_vector, limit=top_k, query_filter=...)
  hits = response.points
  ```
- **教训**:照搬网上老教程的 API 很容易踩版本坑;报 AttributeError 时先 `inspect.signature` 看真实签名。

### 流程教训:提交前务必先看到测试变绿

- 坑 7 那次,我在 `pytest` 结果出来前就 `git commit`,结果提交了红测试,只能补一个 fix commit。
- **规矩**:`run test → 看到 PASS → 才 commit`,顺序不能乱。

## M2 检索与生成阶段

### 坑 9:第一次端到端问答,检索召回的全是"噪音块",答不出已有内容

- **现象**:库里明明有 `tutorial-path-params.md`,但问"FastAPI 里路径参数怎么声明?"
  (限定 `--library fastapi`),模型却回答"根据现有资料无法回答"。
- **排查**:打印检索召回的 Top-4,发现召回的全是
  `FastAPI Cloud is the primary sponsor`、`Deploying to FastAPI Cloud` 这类
  **赞助/部署广告**块,分数挤在 0.64–0.73 几乎没有区分度;真正讲路径参数的那篇根本没进 Top-4。
- **原因**(两个坑叠加,正是 notes/00 预言的):
  1. **切分**:M1 按固定字符窗口硬切,文档里大量营销/导航噪音被切成独立块,淹没了正文。
  2. **检索不准**:中文提问 + 英文文档跨语言,加上 embedding 本身区分度有限,语义匹配捞不到对的块。
- **处理**:M2 **按设计不修**,只如实记录。这正是项目"先用朴素做法把坑暴露出来"的意义,
  留给 M4 评估阶段量化影响、再针对性优化(更好的切分 / 重排 / 查询改写)。
- **亮点**:**防幻觉那一层工作得很好** —— 检索没给对资料时,模型老实拒答而非瞎编。
  这说明 M2 的强约束 prompt(`generate.py` 的三道闸)是有效的。问题出在"检索"不在"生成"。
- **教训**:RAG 答不好,先分清是**检索**问题还是**生成**问题。打印召回的 Top-K 和分数是第一手段;
  这次若不打印,很容易误以为是模型/prompt 不行,实际是根本没检索到。

### 流程沿用:每一步先看测试变绿再继续

- M2 全程 TDD:每个模块先写测试看它失败(RED)、再写最小实现看通过(GREEN),最后跑全量
  27 个测试确认没破坏 M1。沿用了坑7 之后立的规矩:`run test → 看到 PASS → 才 commit`。

## M3 服务化阶段

### 坑 10:uvicorn 启动报 `Address already in use`

- **现象**:`python scripts/serve.py`(默认 8000)报
  `[Errno 48] error while attempting to bind on address ('0.0.0.0', 8000): address already in use`。
- **原因**:8000 端口已被别的进程占用(本机常见:另一个开发服务、上次没关干净的 uvicorn)。
- **解法**:
  - 查谁占着:`lsof -ti:8000`(拿到 PID,必要时 `kill`)。
  - 或直接换端口:`python scripts/serve.py --port 8011`。
- **教训**:服务化后,端口冲突是最常见的启动失败原因;报错先看是不是 `Errno 48`,而不是怀疑代码。
