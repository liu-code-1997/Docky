# 03 · 切分 chunking

## 为什么要切分

RAG 里,embedding 和检索都**以"块(chunk)"为单位**:
- 一个块 → 一个向量 → 存进向量库的一个点。
- 检索时返回的也是"最相关的几个块"。

所以灌数据前,必须先把长文档切成一块一块。切分的好坏**直接决定检索质量**,是 RAG 里最影响效果、也最容易被忽视的一环。

## 我们的第一版:固定字符窗口 + 重叠

最朴素的做法(`src/rag/chunking.py`):

- **chunk_size**:每块最多多少字符(本项目默认 800)。
- **overlap**:相邻两块**重叠**多少字符(本项目默认 150)。

```
原文: [================== 长文本 ==================]
块0:  [----800----]
块1:        [----800----]      ← 与块0 末尾重叠 150
块2:              [----800----]
        ↑ 每次前进 step = chunk_size - overlap = 650
```

代码核心:
```python
step = chunk_size - overlap
start = 0
while start < len(text):
    chunks.append(text[start:start+chunk_size])
    start += step
```

## 三个权衡(也是三个坑)

### 坑 2(头号难题):chunk_size 切多大?

| | 切太大(如 2000) | 切太小(如 100) |
|---|---|---|
| 检索精度 | 差:一块里混了多个话题,向量"语义模糊" | 好:每块话题集中 |
| 上下文完整 | 好:信息齐全 | 差:一句话被切成几块,语义断裂 |
| 塞进 LLM | 占空间,容易超长 | 省空间,但可能信息不全 |

没有万能值,**要靠评估(M4)来调**。本项目先用 800 起步,M4 会实测调它看分数变化。

### 坑 3:embedding 静默截断

如果 chunk_size 设得比 embedding 模型的最大输入还大,超出的部分会被**悄悄丢掉**(见 [[02-embedding与向量]])。所以 chunk_size 不能盲目设大。

### 坑 1:不理解语义边界

我们这版是**纯按字符数**切,它不认识句子、段落、代码块的边界,可能:
- 把一个句子从中间劈开。
- 把一段代码块切成两半 → 检索到半段代码,对技术文档尤其糟糕。

**overlap 的作用**就是缓解这个问题:相邻块留一段重叠,让被切断的语义至少在某一块里是完整的。代价是**冗余存储**(重叠部分存了两次)。

## 更好的切分方式(以后可以学)

第一版故意用最朴素的,好让你看到它的问题。进阶方向(本项目暂不做,YAGNI):
- **按结构切**:按 Markdown 标题、段落、代码块边界切,不破坏语义单元。
- **递归切分**:优先按段落,过长再按句子,再按字符。
- **语义切分**:用模型判断在哪切最合理。

## 实际验收数据

5 篇 FastAPI 文档(约 6 万字符)→ 用 800/150 切 → **91 个块**入库。
你可以改 `config.py` 的 `chunk_size`,重跑 `python scripts/ingest.py`,看块数怎么变。
